# 04_prd_canvas

This folder captures the AI PRD canvas for the aspect sentiment use case.

## Purpose

The purpose of this folder is to define the product clearly before solution design. It helps the team align on the problem, users, scope, expected outputs, and success criteria.

## Why this matters

A good PRD canvas prevents the team from jumping too quickly into model or architecture decisions. It makes sure the business problem and product outcome are clear first. [web:30][web:130]

## Folder contents

| File | Purpose |
|---|---|
| [ai-prd-canvas](ai-prd-canvas.md) | Main product canvas for the use case. |
| [product-summary](product-summary.md) | Short summary of what the product does. |
| [product-vision](product-vision.md) | Long-term vision for the product. |
| [problem](problem.md) | Defines the customer or business problem. |
| [target-users](target-users.md) | Lists the main users and their needs. |
| [scope](scope.md) | States what is in scope and out of scope. |
| [core-aspects](core-aspects.md) | Lists the initial sentiment aspects. |
| [success-metrics](success-metrics.md) | Defines how success will be measured. |
| [assumptions](assumptions.md) | Documents assumptions for the first version. |
| [constraints](constraints.md) | Captures the limits that shape the scope. |
| [non-goals](non-goals.md) | Clarifies what this version will not do. |

## Example use case

For aspect sentiment, the PRD should define a product that can detect themes like delivery, support, pricing, and product quality, and assign sentiment to each of those aspects. [web:56][web:99]

## Output of this folder

By the end of this folder, the team should have:
- a clear product definition,
- a scoped first release,
- and agreed success metrics before moving into solution design.
