## 🚀 Generic Operating Model
MULTI-TEAM OPERATING MODEL
-  ├── 20% CAPACITY: Tech debt + innovation
-  ├── 70% CAPACITY: Committed delivery
-  ├── 10% CAPACITY: Dependencies + surprises
-  └── TARGET: Predictable weekly velocity


- **OKR Cascade**: Align top-level company OKRs (3-5 max) to team-level via shared metrics (e.g., model accuracy → business KPIs). Quarterly reviews ensure 80%+ achievement.
- **Capacity Planning**: Baseline from historical velocity; forecast sprints at 80-90% utilization. Tools: Jira dashboards, MLflow for experiment tracking.
- **Outcome**: 90%+ sprint commitments met; reduced burnout in AI teams handling retraining cycles.

## 02-velocity-scaling
**Team maturity progression | Consistent sprint capacity**

### 📈 Team Maturity Progression (L1→L3)

| Stage | Velocity Stability | Dependencies | Definition of Done      |
|-------|--------------------|--------------|-------------------------|
| **L1** | ±30% variance     | Ad-hoc      | Partial                |
| **L2** | ±10% variance     | Structured sync | Complete           |
| **L3** | ±3% variance      | Automated   | Production-ready       |

- **Progression Path**: Quarterly audits using GitHub benchmarks or DORA metrics. L1: Basic sprints; L2: Cross-training/automation; L3: Autonomous scaling.
- **Scaling Tactics**: Stable velocity over 6-8 sprints before hiring; pair programming for AI complexities like data drift.
- **Outcome**: Consistent capacity for recommendation systems/NLP projects; plateau at sustainable high-performer paces.

## 03-program-governance
**Risk register + planning templates | Zero scope creep**

- **Risk Register**: Centralized (e.g., Confluence/Jira); log AI risks (data drift, bias, latency). Bi-weekly reviews with owners/mitigations.
- **Planning Templates**: Standardized intake forms capturing autonomy levels, ROI thresholds, OKR linkage. Council approval for high-risk changes.
- **Enforcement**: All scope changes tied to OKR impact; rejection rates tracked for zero creep.
