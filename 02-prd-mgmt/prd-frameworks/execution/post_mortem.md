# Post-Mortem Analysis

## Overview

A Post-Mortem is a structured review conducted after:
- incidents
- failures
- launches
- projects

The goal is to identify:
- root causes
- lessons learned
- improvement opportunities

Strong post-mortems focus on:
- learning
not blame.

---

# Core Questions

| Question | Purpose |
|---|---|
| What happened? | Timeline reconstruction |
| Why did it happen? | Root cause analysis |
| What worked well? | Preserve strengths |
| What failed? | Identify gaps |
| What should improve? | Action items |

---

# Typical Structure

## Summary
High-level overview.

## Timeline
Chronological events.

## Impact
Business/customer effects.

## Root Causes
Underlying contributing factors.

## Action Items
Preventive improvements.

---

# Example

## Incident
Checkout outage during sale event.

### Root Cause
Database scaling issue.

### Impact
- Revenue loss
- Customer frustration

### Action Items
- Load testing improvements
- Autoscaling implementation

---

# Best Used When

- Production incidents
- Launch reviews
- Project retrospectives
- Operational failures

---

# Advantages

- Encourages organizational learning
- Improves systems reliability
- Identifies process weaknesses
- Prevents repeated failures

---

# Limitations

- Blame culture reduces effectiveness
- Poor follow-through weakens impact

---

# Best Practices

- Maintain blameless culture
- Focus on systems, not individuals
- Create actionable follow-ups
- Share learnings broadly
