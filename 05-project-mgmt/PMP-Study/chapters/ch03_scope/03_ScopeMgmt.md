## Project Scope Mgmt
 - Includes the processes required to ensure that the project includes all the work required, and only the work required, to complete the project successfully

<img src="images/ScopeMgt.png" alt="ScopeMgt" height="400" width="300">

## 1. Plan scope
Project Scope - The work performed to deliver a product, service, or result with the specified features and functions.
Plan Scope Management process of creating a scope management plan that documents how the
project and product scope will be defined, validated, and controlled.

Completion of project scope is measured against project plan
Completion of product scope is measured against product requirements.
Validate scope is process of formalizing acceptance of completed project deliverables

**Key Benefit**: Direct link between project and organization commitment

**ITTO (Input, Tools & Techniques, Output)**
| Inputs                                      | Tools & Techniques                     | Outputs              |
|--------------------------------------------|----------------------------------------|-----------------------|
| 1. Project Charter                         | 1. Expert judgement                    | 1. Scope Plan         |
| 2. Project Plan                            | 2. Data gathering                      | 2. Requirement plan   |
| 3. EEFs (Enterprise Environmental Factors) | 3. Meetings                            |                       |
| 4. OPAs (Organizational Process Assets)    |                                        |                       |

**Output:**
- Scope Plan: Describes how the scope will be defined, developed, monitored, controlled, and validated.

Assumption Log: Describes how project and product requirements will be analyzed, documented, and managed

**Template:** [Project Scope](../../templates/ProjectScope_template.docx)

## 2. Collect scope
Collect Scope - Process of determining, documenting, and managing stakeholder needs
and requirements to meet objectives

**Key Benefit**: Provides basis for defining the product scope and project scope

**ITTO (Input, Tools & Techniques, Output)**
| Inputs                                      | Tools & Techniques                     | Outputs              |
|--------------------------------------------|----------------------------------------|-----------------------|
| 1. Project Charter                         | 1. Expert judgement                    | 1. Requirement doc    |
| 2. Project Plan, Project docs              | 2. Data gathering                      | 2. Requirement traceability matrix   |
| 3. EEFs (Enterprise Environmental Factors) | 3. Decision Making                     |                       |
| 4. OPAs (Organizational Process Assets)    | 4. Team Skills                         |                       |

**Output:**
- Requirement doc: describes how individual requirements meet the
business need for the project. (Business, stakeholder, solution, project, quality requirements

- Requirement traceability matrix:: Links product requirements from their origin to the deliverables.

**Template:** [Requirement Traceability Matrix](../../templates/RequirementTraceabilityMatrix_template.xlsx)

## 4. WBS scope
Collect Scope - Process of subdividing project deliverables and project work into smaller manageable components

**Key Benefit**: framework of what has to be delivered.

**ITTO (Input, Tools & Techniques, Output)**
| Inputs                                      | Tools & Techniques                     | Outputs              |
|--------------------------------------------|----------------------------------------|-----------------------|
| 1. Scope Plan, Project docs                | 1. Expert judgement                    | 1. Scope baseline    |
| 2. EEFs (Enterprise Environmental Factors) | 3. Decomposition                       | 2. Update project docs  |
| 3. OPAs (Organizational Process Assets)    | 4. Team Skills, Product Analysis       |                       |

   - WBS is a hierarchical decomposition of the total scope of work to be carried to accomplish the project objectives and create the required deliverables
   - work package is the work defined at the lowest level of the WBS for which cost and duration can be estimated and managed

**Output:**
- Scope baseline: Approved version of scope statement

**Template:** [WBS](../../templates/WBS_template.xlsx)

<br>

## 5. Validate scope
Validate Scope - Process of formalizing acceptance of the completed project deliverables

**Key Benefit**: brings objectivity to the acceptance process and increases the
probability of final product, service, or result acceptance by validating each
deliverable.

**ITTO (Input, Tools & Techniques, Output)**
| Inputs                                      | Tools & Techniques                     | Outputs              |
|--------------------------------------------|-----------------------------------------|-----------------------|
| 1. Project Plan, Project docs                | 1. Inspection                           | 1. Accepted deliverables    |
| 2. Verified deliverables                     | 2. Decision Making                      | 2. WPI  |
| 3. WPD                                       |                                         | 3. Change Requests          |


**Output:**
- Accepted Deliverables: Deliverables that meet the acceptance criteria are formally signed off and approved by the customer or sponsor.
- Change Request: Completed deliverables that have not been formally accepted are documented, along with the reasons for non-acceptance of those deliverables. 
- Those deliverables may require a change request for defect repair

<br>

## 6. Control scope
Control Scope - Process of monitoring the status of the project and product
scope and managing changes to the scope baseline

**Key Benefit**: Scope baseline is maintained throughout the project.

**ITTO (Input, Tools & Techniques, Output)**
| Inputs                                      | Tools & Techniques                     | Outputs              |
|--------------------------------------------|-----------------------------------------|-----------------------|
| 1. Project Plan, Project docs                | 1. Variance Analysis                  | 1. Project plan updates, project doc updates   |
| 2. Project docs                              | 2. Trend Analysis                     | 2. WPI  |
| 3. WPD                                       |                                       | 3. Change Requests          |

**Output:**
 - WPI: Work performance information correlates and provides information how the project and product scope are performing compared to the scope baseline.
 - Change Request: Analysis of project performance may result in a change request to the scope and schedule baseline leading to corrective action/preventive action/Defect repairs
 - The uncontrolled expansion to product or project scope without adjustments to time, cost, and resources is referred to as scope creep
